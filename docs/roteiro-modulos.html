Módulo

Ambientes seguros (produção vs. testes)

Problema que resolve: hoje, quando você testa um upload de emergências, o teste apaga e reinsere dados no banco real. Não tem volta. Profissional resolve isso separando o "ambiente de produção" do "ambiente de teste".

Como vai ficar

🟢Produção
Site real + Supabase real
↔
🟡Preview
Site de teste + Supabase de teste

Passo a passo (estimativa: 20 min)

Criar 2º projeto Supabase supabase.com
Acesse o painel → botão New Project → nome obras-dev (ou similar). Plano gratuito permite 2 projetos.
Copiar a estrutura SQL Editor
Rodar os mesmos SQLs (que vamos organizar no Módulo 4) nesse banco novo, criando as mesmas tabelas vazias.
Configurar variáveis no Vercel vercel.com
No projeto, em Settings → Environment Variables, adicionar as chaves do Supabase de teste marcando "Preview" (não "Production"). Assim previews usam o banco de teste; main usa o real. Automaticamente.
Testar
Cada PR seu vai gerar um link de preview que mexe só no banco de teste. Você confere e só depois faz merge para o real.
💡 Você precisa fazer isso agora? Não. Funciona sem. Mas no dia em que você quiser testar com tranquilidade total, este é o caminho.

Organização do banco (Supabase)

Problema que resolve: esta é a maior bagunça hoje. Sua pasta supabase/ tem 10 arquivos SQL com nomes do tipo fix-login.sql, fix-recursion.sql — são "remendos" soltos. Se um dia você precisar recriar o banco do zero (ou criar o ambiente de teste do Módulo 2), não tem ordem clara de qual rodar primeiro.

O conceito: migrações numeradas

Migração ("migration") é cada arquivo SQL que descreve uma mudança no banco. Numerados em ordem, rodados em sequência, recriam o banco do zero de forma confiável.

Hoje (10 arquivos sem ordem)
  • schema.sql
  • auth-setup.sql
  • fix-login.sql
  • fix-recursion.sql
  • fix-emergencias-snapshots.sql
  • fix-sistema-geo-labels.sql
  • emergencias-schema.sql
  • sistema-geo-schema.sql
  • usuarios-internos.sql
  • seed-usuario-marina.sql
Como ficará
  • supabase/migrations/
  •   001_schema_inicial.sql
  •   002_auth_e_perfis.sql
  •   003_emergencias.sql
  •   004_sistemaGeo.sql
  •   005_usuarios_internos.sql
  •   006_timezone_brasil.sql
  • supabase/seeds/
  •   001_usuario_marina.sql
  • supabase/README.md com instruções

Como faço isso (estimativa: ~30 min)

Leio os 10 arquivos atuais
Para entender o que cada um faz e onde os "fixes" se aplicam.
Consolido em arquivos limpos e numerados
Cada migração contém o estado FINAL daquela parte (sem precisar dos "fix" depois).
Escrevo um supabase/README.md
Explicando como rodar tudo do zero num banco novo.
Eu NÃO mexo no seu banco em produção
O banco que está rodando continua intacto. Estes arquivos servem para reconstruir se precisar (ou criar o ambiente de teste do Módulo 2).
✅ Por que este módulo é o mais valioso Sem isso, seu banco é "caixa-preta": existe, funciona, mas ninguém sabe recriar. Depois disso, qualquer banco novo (teste, backup, migração para outra conta) é só rodar os arquivos em ordem.

Módulo

Qualidade automática (ESLint + Prettier)

Problema que resolve: hoje, o estilo do código depende de quem escreveu. Espaços, aspas, vírgulas: cada arquivo um pouco diferente. Profissional usa duas ferramentas automáticas que padronizam tudo.

O que cada ferramenta faz

FerramentaFunçãoExemplo
ESLintDetecta possíveis erros e más práticas (variável não usada, comparação suspeita...)Avisa: "essa variável foi criada e nunca usada"
PrettierReformata o visual do código (espaços, aspas, quebras de linha)Transforma const a={x:1} em const a = { x: 1 }

O que muda no seu dia a dia

  • Salvar o arquivo formata automaticamente.
  • Antes de cada commit, ferramentas rodam sozinhas e bloqueiam código mal formatado.
  • Todo arquivo do projeto fica com o mesmo "sotaque".
💡 Custo Para você, zero — eu instalo e configuro. Para o projeto, adiciona ~2 dependências de desenvolvimento (não afetam o site final).

Fluxo de trabalho profissional

Problema que resolve: "tô meio perdido em quando faço merge, quando crio branch, o que escrever no commit". Aqui vira hábito.

A regra de ouro (ciclo de uma mudança)

1️⃣Você pede algo
→
2️⃣Eu trabalho em branch
→
3️⃣Eu faço push
→
4️⃣Você revisa o PR
→
5️⃣Você faz merge
→
6️⃣Vercel publica

Convenção de mensagens de commit

Pequena regra que profissionaliza o histórico: cada commit começa com um "tipo".

feat: nova funcionalidade
fix: correção de bug
docs: só documentação
refactor: reorganiza código sem mudar comportamento
style: formatação, espaços, vírgulas (sem mudar lógica)
chore: tarefas de manutenção, dependências
perf: melhora de performance

Olhe o histórico recente do seu projeto — eu já estou seguindo isso:

feat: pop-ups exigem confirmação manual (Ok) antes de fechar
fix: elimina re-download desnecessário após upload de emergências
docs: adiciona CLAUDE.md e guia visual interativo

Boas práticas de branch

  • Uma branch por tarefa (não acumular mudanças sem relação).
  • Nome descritivo (ex.: refatora-banco, fix-upload-emergencias).
  • Vida curta: mergeie em poucos dias, não deixe galhos antigos esquecidos.
  • main intocada: tudo passa por PR e merge.
