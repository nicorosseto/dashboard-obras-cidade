Analogia simples Pense numa cozinha de restaurante: o Claude Code é o cozinheiro que prepara os pratos (código); o GitHub é a despensa organizada que guarda todas as receitas e suas versões anteriores; o Vercel é o salão onde os clientes são servidos (o site no ar); e o Supabase é o estoque de ingredientes e a lista de reservas (os dados e os logins).

O ciclo de uma mudança (passo a passo)

#O que aconteceOnde
1Você pede uma alteração e eu escrevo o códigoClaude Code
2Eu faço um "commit" (salvo uma versão) e "push" (envio)Claude Code → GitHub
3Você revisa e faz o "merge" (aprova a mudança)GitHub
4O Vercel percebe a mudança e republica o site sozinhoVercel (automático)
5Usuários acessam o site novo, que lê dados do bancoVercel ↔ Supabase

🌐 Ambientes — a "cópia segura" que você pediu

Você quis fazer uma cópia para mexer sem medo. Boa notícia: você já tem isso. Deixa eu explicar o conceito profissional.

Os 3 tipos de ambiente

AmbienteO que éQuem usa
🟢 ProduçãoO site real, no ar, estávelOs usuários de verdade
🟡 Preview / HomologaçãoUma cópia para testar antes de publicarVocê, para conferir
🔵 DesenvolvimentoOnde a mudança está sendo construídaVocê + Claude Code

Como isso já funciona no seu projeto (branches)

Uma branch ("galho", em inglês) é uma linha paralela de trabalho. É a sua cópia segura: você mexe no galho sem tocar no tronco principal.

main  ← o tronco. É o que está em PRODUÇÃO (site no ar)
claude/sleepy-mccarthy…  ← seu galho de trabalho. Mexemos aqui à vontade
Quando o galho está pronto, você faz o merge (junta ao tronco) e só então a produção muda.
✅ O front-end (site) já está protegido O Vercel cria automaticamente um endereço de Preview para cada galho. Ou seja: você consegue ver suas mudanças num link de teste antes de afetar o site real. Custo: zero.
⚠️ Atenção com o banco de dados (Supabase) O banco é compartilhado: tanto a produção quanto o preview falam com o MESMO Supabase. Então um teste que apaga dados (como o upload de emergências, que faz DELETE) afeta os dados reais. Para segurança total, o profissional cria um 2º projeto Supabase só para testes (o plano grátis permite 2 projetos).
💡 Recomendação para você Você não precisa recriar o projeto do zero. O caminho profissional é: manter a produção no main, fazer toda a reorganização no galho (sem afetar ninguém), testar no link de preview do Vercel e só depois fazer o merge. O galho é a sua cópia integral.

O que cada pasta e arquivo significa. Não precisa decorar — é só para consultar.

dashboard-obras-cidade/ ├── src/ # todo o código do site │ ├── App.jsx # a "espinha dorsal": navegação e layout │ ├── main.jsx # liga o React à página │ ├── index.css # estilos globais │ ├── components/ # peças reutilizáveis da interface │ │ ├── charts/ # gráficos e mapas │ │ └── pages/ # conteúdo de cada aba do painel │ ├── pages/ # telas principais (Home, Login) │ ├── lib/ # lógica sem visual (cálculos, conexões) │ │ ├── supabase.js # conexão com o banco │ │ ├── auth.js # login e logout │ │ └── aggregations.js# contas, KPIs, formatação de datas │ └── data/ # dados fixos (mapa de SP) ├── supabase/ # scripts do banco (HOJE bagunçado — vamos organizar) ├── public/ # arquivos servidos como estão (logos) ├── CLAUDE.md # a "memória" do projeto (recém-criada!) ├── package.json # lista de dependências e comandos └── tailwind.config.js # cores e tema
Termo: dependência São "peças prontas" feitas por outras pessoas que seu projeto usa (React, Recharts, etc.). Ficam listadas no package.json. O comando npm install baixa todas.

📖 Glossário — clique para abrir

Aqueles termos que apareceram e você não fazia ideia do que eram. Aqui, em português claro.

A pasta do seu projeto inteira, com todo o histórico, guardada no GitHub. "Repo" é o apelido.

Uma "fotografia" salva do código num momento. Cada commit tem uma mensagem dizendo o que mudou. Permite voltar no tempo.

Push = enviar seus commits para o GitHub. Pull = trazer mudanças do GitHub para você. É a sincronização nos dois sentidos.

Uma linha de trabalho paralela. Permite mexer no projeto sem afetar a versão principal (main). Sua cópia segura.

Juntar as mudanças de um galho de volta ao tronco (main). É o momento em que o que você testou vira oficial.

Um "pedido de merge" no GitHub. Mostra exatamente o que mudou e permite revisar antes de aprovar. É onde você confere meu trabalho.

Publicar o site. Cada vez que o Vercel pega o código e coloca no ar, ele fez um "deploy".

Front-end = o que o usuário vê e clica (as telas, em React). Back-end = os bastidores (banco de dados, login) — no seu caso, o Supabase.

Regras no banco que controlam, linha por linha, quem pode ver ou alterar cada dado. É o que impede um usuário comum de ver dados que não deveria.

Um arquivo SQL numerado que descreve uma mudança no banco. Aplicados em ordem, recriam o banco do zero de forma confiável. Vamos organizar os seus SQLs assim.

Configurações secretas (como chaves de acesso) que NÃO vão no código. Ficam num arquivo .env.local (escondido do GitHub) e nas configurações do Vercel.

Linter (ESLint) avisa sobre possíveis erros. Formatador (Prettier) deixa todo o código com o mesmo estilo automaticamente. Profissionaliza e evita brigas de espaçamento.

Um arquivo de texto que eu (Claude Code) leio no começo de cada conversa. É a "memória" do projeto: explica o sistema, as regras e as decisões. É como o conhecimento "viaja" entre conversas diferentes.

🚀 Nosso roteiro — o curso em módulos

Vamos um módulo por vez, sem pressa. Cada um deixa o projeto mais profissional. Você decide o ritmo.

0

Fundamentos e orientação ✓ feito

Este guia + o arquivo CLAUDE.md. Você entende as ferramentas e como tudo se conecta.

1

Memória do projeto ✓ feito

Criamos o CLAUDE.md e este guia. Falta um README.md decente (a "capa" do projeto no GitHub).

2

Ambientes seguros

Entender e usar produção vs. preview. Opcional: criar um 2º Supabase só para testes, para nunca arriscar dados reais.

3

Organização do código

Padronizar nomes de pastas e arquivos, separar bem as responsabilidades, eliminar duplicações.

4

Organização do banco (Supabase)

Transformar os 10 SQLs soltos da pasta supabase/ em migrações numeradas e ordenadas.

5

Qualidade automática

Instalar ESLint + Prettier para o código se corrigir e padronizar sozinho. Revisar tratamento de segredos.

6

Fluxo de trabalho profissional

Rotina saudável de branches, commits e Pull Requests. Como revisar e aprovar mudanças com confiança.
